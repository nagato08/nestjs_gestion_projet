1️⃣ Introduction au Projet

Ce projet est une application NestJS full-stack pour la gestion des projets et des tâches. Il est conçu pour permettre aux équipes de collaborer efficacement en organisant les projets, en assignant des tâches, en suivant le temps, et en communiquant en temps réel.

Objectifs Principaux

  • ✅ Gérer les projets avec des statuts et des priorités
  • ✅ Créer et assigner des tâches avec dépendances
  • ✅ Authentifier les utilisateurs avec JWT
  • ✅ Suivre le temps passé sur les tâches
  • ✅ Communiquer en temps réel avec WebSockets
  • ✅ Gérer les documents et versions
  • ✅ Envoyer des notifications
  • ✅ Utiliser l'IA pour la planification

Public Cible

Débutants en NestJS qui souhaitent comprendre comment construire une application d'entreprise moderne avec authentification, base de données relationnelle, WebSockets et architecture modulaire.

2️⃣ Architecture & Stack Technologique

NestJS est un framework modulaire et extensible construit sur Express.js. Il utilise TypeScript et suit une architecture inspirée d'Angular.

Framework Principal: NestJS v11.0.1
Runtime: Node.js

🔐 AuthModule - Authentification

Responsabilité: Gérer l'inscription, la connexion et la sécurité des utilisateurs.

Fichiers Clés:

  • auth.controller.ts - Routes POST /register, POST /login, GET /profile, GET /users, DELETE /:id
  • auth.service.ts - Logique de validation, hachage JWT avec bcrypt
  • jwt.strategy.ts - Stratégie de validation JWT
  • guards/jwt-auth.guard.ts - Protège les routes avec @UseGuards(JwtAuthGuard)
  • guards/roles.guard.ts - Contrôle d'accès par rôle avec @Roles(Role.ADMIN)
  • auth.module.ts - Enregistre JwtModule et PassportModule

Processus de Connexion:

Utilisateur envoie email + password ↓ AuthService.login() valide les credentials ↓ Hachage du password comparé avec bcrypt.compare() ↓ JWT généré avec jwtService.signAsync(payload) ↓ Retour au client: {user, access_token, dashboardUrl}

Responsabilité: CRUD sur les projets, gestion des membres, codes d'accès.

Entités Principales:

6️⃣ Authentification & Sécurité

Flux d'Authentification JWT

┌─────────────────────────────────────┐ │ 1. Client envoie credentials │ │ POST /auth/login │ │ { email, password } │ └─────────────────────────────────────┘ ↓ ┌─────────────────────────────────────┐ │ 2. AuthService.login() vérifie │ │ - Email existe? │ │ - Password valide? (bcrypt) │ └─────────────────────────────────────┘ ↓ ┌─────────────────────────────────────┐ │ 3. JWT généré avec payload: │ │ { sub, email, role, department } │ │ Signé avec JWT_SECRET │ │ Expire dans 30 jours │ └─────────────────────────────────────┘ ↓ ┌─────────────────────────────────────┐ │ 4. Client reçoit access_token │ │ Stocke dans localStorage │ └─────────────────────────────────────┘ ↓ ┌─────────────────────────────────────┐ │ 5. Requêtes futures │ │ Authorization: Bearer <token> │ │ JwtAuthGuard valide le token │ └─────────────────────────────────────┘

Rôles & Contrôle d'Accès

Rôle Permissions
ADMIN ✅ Voir tous les utilisateurs
✅ Supprimer les utilisateurs
✅ Accès complet
PROJECT_MANAGER ✅ Créer/modifier les projets
✅ Assigner les tâches
✅ Voir les rapports
EMPLOYEE ✅ Voir ses projets
✅ Accepter les tâches
✅ Voir ses tâches

Protections Appliquées

🔐 JwtAuthGuard - Valide la présence du JWT dans les headers
🛡️ RolesGuard - Vérifie que le rôle fait partie de @Roles()
🔑 bcrypt - Hachage 10 rounds pour les passwords (jamais stocké en clair)
⏰ Token Expiry - 30 jours pour éviter les vols à long terme
🚫 Soft Delete - Users/Projects marqués deletedAt au lieu d'être supprimés
🌐 CORS - Contrôle des origines autorisées
⛔ Throttle - Limite 10 requêtes par minute par IP

Données Sensibles

Les données suivantes ne sont JAMAIS renvoyées au client:

  • Password (haché en base)
  • resetPasswordToken
  • twoFactorSecret

7️⃣ Flux de Données Principaux

Flux 1: Création d'un Projet

Frontend appelle: POST /project { name, description, startDate, ... } Authorization: Bearer <JWT> ↓ ProjectController.createProject() ↓ ProjectService.createProject(userId, dto) - Valide l'DTO avec les pipes - Génère un projectCode unique - Utilise une TRANSACTION Prisma: • Crée le Project • Ajoute owner comme ProjectMember • Crée une Conversation pour le chat ↓ Retour au client: { id, name, projectCode, owner, createdAt }

Flux 2: Créer & Assigner une Tâche

POST /tache { title, projectId, assignedUserIds: [userId1, userId2], priority, deadline } ↓ TacheService.createTask() - Vérifie que l'utilisateur est membre du projet - Valide que les assignés sont aussi membres - Crée la Task avec statut TODO ↓ Pour chaque assigné: - Crée une TaskAssignment (M:N) - Notifie l'utilisateur ↓ Retour: { id, title, assignments: [...] }

Flux 3: Mettre à Jour le Statut d'une Tâche

PATCH /tache/:taskId/status { status: "DOING" } ↓ Vérifie les dépendances: - S'il y a des tâches qui la bloquent → Erreur "Task is blocked by..." - Sinon, change le statut ↓ Notifie tous les assignés ↓ WebSocket broadcast: emit('task_updated', { taskId, newStatus }) Tous les clients reçoivent la MAJ

Flux 4: Communication Temps Réel (Chat)

Client A se connecte au WebSocket: socket.io connection → verifyJWT() ↓ Client A rejoint une room: socket.emit('join_project', projectId) ↓ Client A envoie un message: socket.emit('send_message', { content: "Besoin d'aide" }) ↓ ChatService.createMessage() sauvegarde en DB ↓ ChatMessage est broadcasté: io.to(projectRoom).emit('message_received', msg) Tous les clients dans la room reçoivent le message

8️⃣ Base de Données (Prisma & PostgreSQL)

Qu'est-ce que Prisma?

Prisma est un ORM (Object-Relational Mapping) qui transforme les tables SQL en objets TypeScript type-safe. Au lieu d'écrire du SQL brut, vous utilisez une API JavaScript élégante.

Schéma Prisma (prisma/schema.prisma)

Définit les modèles de données et leurs relations:

model User { id String @id @default(cuid()) email String @unique password String firstName String @map("first_name") lastName String @map("last_name") role Role @default(EMPLOYEE) // Relations projectsOwned Project[] @relation("ProjectOwner") projectMembers ProjectMember[] taskAssignments TaskAssignment[] createdAt DateTime @default(now()) updatedAt DateTime @updatedAt } model Project { id String @id @default(cuid()) name String description String? ownerId String owner User @relation("ProjectOwner", fields: [ownerId], references: [id]) members ProjectMember[] tasks Task[] deletedAt DateTime? // Soft delete } model Task { id String @id @default(cuid()) title String projectId String project Project @relation(fields: [projectId], references: [id]) assignments TaskAssignment[] // Dépendances dependencies TaskDependency[] @relation("BlockingTask") blockedBy TaskDependency[] @relation("BlockedTask") } enum Role { ADMIN PROJECT_MANAGER EMPLOYEE } enum TaskStatus { TODO DOING DONE } enum Priority { LOW MEDIUM HIGH CRITICAL }

Opérations Prisma Courantes

Opération Exemple
CREATE prisma.user.create({ data: {...} })
READ prisma.user.findUnique({ where: {id} })
UPDATE prisma.user.update({ where, data })
DELETE prisma.user.delete({ where })
FIND MANY prisma.task.findMany({ where, include, orderBy })
TRANSACTION prisma.$transaction(async tx => {...})

Migrations - Évolution du Schéma

Chaque changement du schéma génère une migration SQL:

🔟 Concepts Clés NestJS

1. Modules (@Module)

Les modules sont des conteneurs qui regroupent contrôleurs et services liés. Chaque module peut importer d'autres modules et exporter ses providers.

@Module({ imports: [ JwtModule.register({...}), PassportModule ], controllers: [AuthController], providers: [AuthService, JwtStrategy, PrismaService], }) export class AuthModule {}

2. Contrôleurs (@Controller)

Les contrôleurs reçoivent les requêtes HTTP et appellent les services. Ils utilisent des décorateurs pour définir les routes.

@Controller('auth') export class AuthController { constructor(private authService: AuthService) {} @Post('login') @ApiOperation({ summary: 'Connexion' }) async login(@Body() loginDto: LoginDTO) { return this.authService.login(loginDto); } }

3. Services (@Injectable)

Les services contiennent la logique métier. Ils sont injectables dans les contrôleurs et autres services.

@Injectable() export class AuthService { constructor( private prisma: PrismaService, private jwtService: JwtService ) {} async login(dto: LoginDTO) { // Logique métier ici } }

4. DTOs (Data Transfer Objects)

Les DTOs valident et typent les données entrantes. Utilisent class-validator et class-transformer.

export class CreateUserDto { @IsEmail() email: string; @MinLength(8) password: string; @IsIn([Role.ADMIN, Role.EMPLOYEE]) role: Role; }

5. Guards (@UseGuards)

Les gardes contrôlent l'accès aux routes avant que le contrôleur ne s'exécute.

@UseGuards(JwtAuthGuard, RolesGuard) @Roles(Role.ADMIN) @Get('users') async getUsers() { // Seulement les ADMIN peuvent accéder }

6. Pipes (@UsePipes)

Les pipes transforment et valident les données avant qu'elles n'atteignent le contrôleur.

@Post() @UsePipes(new ValidationPipe({ whitelist: true })) async create(@Body() dto: CreateProjectDto) { // Les données sont validées et transformées }

7. Decorateurs Personnalisés

Créez des décorateurs réutilisables pour extraire des données spécifiques.

@Decorator() export const CurrentUser = createParamDecorator( (data, ctx) => { const req = ctx.switchToHttp().getRequest(); return req.user; } ); // Utilisation dans le contrôleur: @Get('profile') getProfile(@CurrentUser() user) { // user contient les infos JWT }

8. Gateways WebSocket

Les gateways gèrent la communication temps réel avec Socket.io.

@WebSocketGateway({ cors: { origin: '*' } }) export class AppGateway implements OnGatewayConnection { @SubscribeMessage('join_project') async joinProject(client: Socket, projectId: string) { client.join(`project-${projectId}`); } @SubscribeMessage('send_message') async sendMessage(client: Socket, message: any) { this.server.emit('message_received', message); } }

9. Exception Filters

Les filtres centralisent la gestion des erreurs.

@Catch() export class AllExceptionsFilter implements ExceptionFilter { catch(exception: unknown, host: ArgumentsHost) { // Traiter l'exception globalement } }

10. Injection de Dépendances

NestJS gère automatiquement les dépendances. Déclarez simplement dans le constructeur.

export class ProjectService { constructor( private prisma: PrismaService, private notificationService: NotificationService, private logger: Logger ) {} }
